Getting Started

Overview

The Gamma Iota Lambda Chapter website is a static HTML website. That means there is no WordPress, no Squarespace dashboard, no database, and no software to log into. The entire site is made up of plain text files — specifically .html, .css, and a single shared style.css — that GitHub serves directly to visitors.

If you've never maintained a website before, here is the core idea: to change what people see on the site, you edit the text files and push them to GitHub. GitHub Pages (a free hosting service) automatically publishes whatever is in the brooklyn-alphas repository.

Key Principle
This site has no "admin panel." Every change — updating board members, adding event photos, changing text — is made by editing the actual code files. This sounds technical, but most updates follow simple, repeatable patterns that are fully documented here.

What the site does:

Homepage
Hero section, chapter intro, programs overview, meetings info, contact form, and Substack newsletter embed.
Executive Board
Headshot grid of the current term's board members with names and roles.
Events
Photo carousels for recent chapter events, loaded automatically from photo folders.
Alpha Academy
Program description for the chapter's signature youth leadership initiative.
Scholarship
Judge Myles A. Paige Scholarship info, application link, and deadline.
Histories
Full chapter history (with timeline) and Alpha Phi Alpha fraternity history.

Getting Started

Tools You Need

Before you can make and publish changes, you need a few things set up on your computer. You only need to do this once.

1
A GitHub account. Create one at github.com if you don't have one. Then ask the current webmaster to add you as a collaborator on the Gamma-Iota-Lambda/brooklyn-alphas repository.
2
Git. This is the software that syncs files between your computer and GitHub. Download from git-scm.com. On a Mac, you may already have it — type git --version in your Terminal to check.
3
A code editor. Visual Studio Code (VS Code) is free and the most beginner-friendly option. Download from code.visualstudio.com. It gives you color-coded HTML, a built-in file explorer, and a terminal — everything in one place.
4
Clone the repository. Open your Terminal (Mac) or Command Prompt (Windows), navigate to where you want the files, and run:

git clone https://github.com/Gamma-Iota-Lambda/brooklyn-alphas.git

This downloads the entire site to a folder called brooklyn-alphas on your machine.
5
Node.js (for image tools only). If you plan to add event photos or compress images, install Node.js from nodejs.org. Then inside the project folder, run npm install to set up the helper scripts. You can skip this step if you're only making text edits.

Once cloned, open the brooklyn-alphas folder in VS Code. You'll see the full file list on the left side. Double-click any file to open and edit it.

Making Edits

Editing Text & Content

All text on the site is written directly in HTML files. HTML uses tags — words wrapped in angle brackets like <p> and <h2> — to mark up content. You don't need to know how to build HTML from scratch; you just need to recognize the patterns and know how to change the text inside the tags.

Here are the tags you'll encounter most often:

<p>
A paragraph of body text. Most of the readable content on each page is in <p> tags.
<h1> – <h3>
Headings of decreasing size. The big headline on a page is usually an <h1>.
<a href="...">
A link. The URL goes inside the quotes after href=. The clickable text goes between the opening and closing tags.
<img src="..." alt="...">
An image. The file path goes in src=. Always keep a meaningful description in alt= for accessibility.
<strong>
Bold text within a paragraph.

Example — changing a paragraph: Open alpha-academy.html, find the text you want to update using VS Code's Find feature (Ctrl+F / Cmd+F), and edit the text inside the <p> tags. Do not delete the opening <p> or closing </p>.

Before
<p>Alpha Academy is our Signature Leadership and College Readiness program.</p>
After
<p>Alpha Academy is our award-winning Leadership and College Readiness program.</p>
Important
Always change only the text content between tags — never delete or modify the tags themselves or the class names, IDs, or attributes on them, unless you specifically know what they do. If you break a tag, the page may display incorrectly.

Using Find & Replace in VS Code: Press Ctrl+H (Windows) or Cmd+H (Mac) to open the Find & Replace panel. Type the old text in the top box and the new text in the bottom box. This is especially useful for updating things like a year across the whole file.

Styling & CSS

All visual design — colors, fonts, spacing, layout — is controlled by CSS (Cascading Style Sheets). The site uses a single shared stylesheet called style.css in the root of the repo, plus small <style> blocks inside each page's <head> for page-specific styles.

Where to edit: If a change should affect every page (nav, footer, buttons, base typography), edit style.css. If it only affects one page, edit the <style> block in that page's <head>.

CSS Variables — The site uses a system of design tokens defined at the top of style.css. Instead of hardcoding color values everywhere, any element that uses gold references var(--gold). This means changing the brand gold color is a one-line edit:

style.css — Color variables (top of file)
/* ─── VARIABLES ─── */
:root {
  --gold:        #B8860B;   /* primary gold — used for accents */
  --gold-bright: #D4A017;   /* brighter gold — buttons, highlights */
  --gold-light:  #FAF0BE;   /* pale gold — rarely used */
  --black:       #0A0A0A;   /* near-black background */
  --black-soft:  #111111;   /* slightly lighter background */
  --white:       #FFFFFF;
  --brown-deep:  #7E4300;   /* warm brown — rare accent */
}

Fonts — The site uses three typefaces, all loaded from Google Fonts:

Playfair Display
Serif — used for large display headings, hero titles, section names.
Barlow Condensed
Condensed sans — used for labels, eyebrows, nav links, buttons, uppercase small text.
Barlow
Regular sans — used for body copy, paragraphs, general text.
Tip
The style.css file is 1,471 lines long. Use Ctrl+F in VS Code to search for the class name you want to change, rather than scrolling through the whole file. Comments like /* ─── NAV ─── */ throughout the file act as labeled sections to help you navigate.

Events & Media

Events Page

The events page (events.html) displays photo carousels for recent chapter events. Each event gets its own section with a photo carousel, title, date, and description.

Events are organized in the assets/ folder. Each event has its own subfolder containing all of its photos plus a photos.json manifest file that the page uses to automatically load and display them.

How the assets/ folder is organized
assets/ ├── mlk-2025/ ← MLK Day of Service 2025 │ ├── photos.json ← auto-generated index of photos │ ├── photo-001.jpg │ └── photo-002.jpg ├── roots-and-proofs-2026/ │ ├── photos.json │ └── ... photos ... └── chapter-life/

To add a new event:

1
Create a folder in assets/. Name it in lowercase with hyphens, using the event name and year: chapter-picnic-2026, step-show-2026, etc. The folder name becomes the event ID referenced in the HTML.
2
Add photos to the folder. Compress them first (see Section 11). Name them sequentially for best display order: photo-001.jpg, photo-002.jpg, etc.
3
Generate the manifest. Open a Terminal in the repo folder and run: node generate-manifests.js. This scans every event folder and creates a photos.json file in each one. To generate just one event: node generate-manifests.js chapter-picnic-2026.
4
Add an event section in events.html. Copy an existing event block (an ev-event div) and update the event title, date, description, and the data-event attribute to match your new folder name.
Event section template in events.html
<section class="ev-event">
  <div class="ev-event-meta">

    <div class="ev-event-date">July 12, 2026</div>
    <p class="ev-event-desc">Brothers and families gathered ...</p>
  </div>
  <div class="ev-carousel" data-carousel
       data-event="chapter-picnic-2026">  <!-- must match folder name -->
    <div class="ev-carousel-stage"></div>
    <div class="ev-carousel-caption">
      <span class="ev-caption-text"></span>

        <span><span class="ev-current">1</span> / <span class="ev-total"></span></span>
        <button class="ev-next">›</button>
      </div>
    </div>
  </div>
</section>

Events & Media

Photo Carousels

The photo carousels on events.html are manifest-driven — meaning the page doesn't have photos hardcoded in the HTML. Instead, when the page loads, it reads the photos.json file from each event folder and builds the carousel dynamically from that list.

The photos.json file looks like this:

Example: assets/mlk-2025/photos.json
{
  "event": "mlk-2025",
  "count": 3,
  "photos": [
    { "file": "photo-001.jpg", "caption": "Brothers preparing meal bags" },
    { "file": "photo-002.jpg", "caption": "Distribution at Atlantic Men's Shelter" },
    { "file": "photo-003.jpg", "caption": "Chapter photo after the event" }
  ]
}

Adding captions: After running node generate-manifests.js, open each photos.json file and fill in the "caption" fields. They default to empty strings. Re-running the manifest generator will preserve any captions you have already written — it only adds new photos, it doesn't overwrite.

Carousel behavior: Carousels auto-advance every 4 seconds. They pause when hovered, focused with a keyboard, or scrolled out of view. Visitors can also swipe left/right on mobile, use the prev/next buttons, or click the thumbnail strip. They are fully keyboard-accessible.

Photo Tips
Portrait-oriented photos (taller than wide) use object-fit: contain to avoid cropping. Landscape photos fill the frame with object-fit: cover. Either orientation works, but a consistent orientation per event looks best.

Image Compression

Large uncompressed images slow down the site significantly. The repo includes a script called compress.js that automatically compresses all images using a tool called sharp. Before running it, you need Node.js installed (see Section 3).

1
Install dependencies (first time only). In a Terminal, navigate to the repo folder and run: npm install. This installs sharp.
2
Run the compressor. To compress everything: node compress.js. To compress a single folder only: node compress.js headshots/ or node compress.js assets/mlk-2025/.
3
Check the output. The script prints a table showing the original size, compressed size, and % reduction for each file. It automatically backs up originals to an originals/ subfolder inside each directory before compressing — so you can always recover them.
What It Does
The script compresses JPGs and PNGs to 82% quality (a sweet spot where quality loss is invisible) and caps maximum image width at 1,600px. originals/ backup folders are excluded from Git (via .gitignore), so they stay on your machine but don't bloat the repository.

Manual alternative: If you don't have Node.js, you can compress individual images at squoosh.app in a browser. Use the MozJPEG encoder at 80–85% quality, or OxiPNG for PNG files.

SEO & Meta Tags

SEO (Search Engine Optimization) is how the site gets found on Google. Every page is set up with a full suite of meta tags. When you add a new page or significantly update an existing one, you should update these tags to keep them accurate.

Each page's <head> contains the following tags you may need to update:

<title>
The browser tab title and the blue headline in Google search results. Keep it under 60 characters. Include the page topic and "Gamma Iota Lambda."
meta description
The snippet of text shown under the title in Google results. Keep it under 155 characters. Write it like a one-sentence summary for a human reader.
link rel="canonical"
Tells Google the definitive URL for this page. Must match the actual URL exactly, e.g. https://www.gil1906.org/scholarship.html.
og:title, og:description
The title and description used when the page is shared on Facebook, LinkedIn, iMessage, etc. Can differ slightly from the <title> and meta description.
og:image
The preview image shown in social shares. Points to og-image.jpg by default. The ideal size for this image is 1200×630px.
sitemap.xml
A list of all URLs on the site, submitted to Google. Update the <lastmod> date whenever a page changes, and add a new <url> entry whenever you add a new page.
Adding a New Page
When adding a brand new page to the site: (1) copy the full <head> section from an existing page, (2) update every meta tag to reflect the new page, (3) add the page to the nav on every existing page, (4) add a <url> entry in sitemap.xml.

Deployment

Publishing Changes

Publishing changes to the live site is done through Git and a short GitHub review process. Because the main branch is protected — meaning no one can push directly to it — every change goes through a Pull Request (PR). Think of it as a built-in second pair of eyes before anything goes live.

The workflow is: pull latest → create a branch → edit files → commit → push branch → open a PR on GitHub → get it merged. It sounds like more steps than it is — once you've done it once, it takes about two extra minutes.

What is a branch?
A branch is your own private copy of the codebase to work in. Think of main as the published, official version of the site. When you create a branch, you're saying "I want to make some changes without touching the live site yet." Your branch is invisible to visitors. Only when your branch is reviewed and merged into main does anything change on the live site. If something goes wrong on your branch, main is untouched.

Step-by-step: the full publish workflow

1
Pull the latest version of main. Before you start any work, make sure your local copy is up to date. This prevents conflicts if someone else made a change since you last worked on the site.

git checkout main
git pull
2
Create a new branch. Name it something short and descriptive. Use lowercase and hyphens.

git checkout -b update/exec-board-2026-2027

You are now on your new branch. Any changes you make stay here until you open a PR.
3
Make your edits. Open files in VS Code and make your changes as normal. Preview them in a browser by dragging the HTML file into a browser window (or right-click → Open With → Browser) before moving on.
4
Stage and commit your changes.

git add .
git commit -m "update: exec board 2026-2027 term"

You can make multiple commits on the same branch as you work — each commit is a saved checkpoint.
5
Push your branch to GitHub. The first time you push a new branch, use the full command:

git push -u origin update/exec-board-2026-2027

After the first push, you can just use git push for any additional commits on that same branch.
6
Open a Pull Request on GitHub. After pushing, go to github.com/Gamma-Iota-Lambda/brooklyn-alphas. GitHub will show a yellow banner saying your branch was recently pushed — click "Compare & pull request." Add a short description of what you changed and click "Create pull request."
7
Wait for review and merge. Someone with write access to main will review the PR and merge it. Once merged, GitHub Pages automatically rebuilds the site. Changes are typically live within 1–3 minutes of the merge.
Terminal — complete publish workflow at a glance
# Start: make sure you're on main and up to date
git checkout main
git pull

# Create and switch to a new branch
git checkout -b update/exec-board-2026-2027

# ... make your edits in VS Code ...

# Check what you changed
git status

# Stage everything and commit
git add .
git commit -m "update: exec board 2026-2027 term"

# Push your branch (first time on this branch)
git push -u origin update/exec-board-2026-2027

# Then open a Pull Request on GitHub — see Step 6 above
Branch Naming Convention
Name branches after what they do. Good patterns: update/exec-board-2027, feat/chapter-picnic-photos, fix/scholarship-deadline. Keep them lowercase with hyphens. Use a prefix (update/, feat/, fix/) that matches your commit message prefix.
Commit Message Convention
The project uses the prefix feat: for new content, fix: for corrections, and update: for routine updates. Good examples: "feat: add chapter picnic 2026 event photos", "fix: correct scholarship deadline date", "update: exec board 2026-2027".
Before You Push
Always preview the changed page in your browser before pushing. Catching a broken tag locally is much easier than reverting a PR after the fact.

If you accidentally break something: If you haven't pushed yet, run git checkout -- filename.html to restore that file to its last committed state. If you've already pushed but the PR hasn't been merged, you can push a correcting commit to the same branch — it will update the PR automatically. If the PR has already been merged, contact whoever manages the repo to revert the merge.

Troubleshooting

Common issues and how to fix them:

A headshot isn't showing up — I just see the chapter crest placeholder.
Check three things: (1) The filename in the <img src="..."> attribute exactly matches the actual filename in headshots/, including the extension (.jpg vs .jpeg). File names are case-sensitive on the server. (2) The placeholder class is not present on the eb-card-img-wrap div — if it is, remove it. (3) The file was actually pushed to GitHub (run git status to check).
An event carousel is empty — no photos appear.
The carousel reads from assets/[event-name]/photos.json. Check that: (1) the data-event attribute on the .ev-carousel div exactly matches the folder name in assets/ — case matters. (2) photos.json exists in that folder — if not, run node generate-manifests.js. (3) The photos listed in photos.json actually exist in the folder. (4) All files were pushed to GitHub.
I pushed a change but the site doesn't look different.
Your browser is likely showing a cached version of the page. Open the site in a private/incognito browser window, or do a hard refresh: Ctrl+Shift+R (Windows) / Cmd+Shift+R (Mac). Also check GitHub (go to the repo and look at the file) to confirm your push went through. GitHub Pages can sometimes take 1–3 minutes to update.
The mobile menu isn't working or the page looks broken on mobile.
Open the browser's developer tools (F12 or right-click → Inspect) and check the Console tab for red error messages. Usually this means a JavaScript error caused by a mistyped tag or a missing closing tag somewhere in the HTML. The error message will tell you which line to look at. You can also use the validator at validator.w3.org — paste in the page's URL and it will highlight any broken HTML.
I accidentally deleted a section of the page while editing.
If you haven't pushed the change yet, run git checkout -- [filename] in the Terminal to restore the file to its last committed state. This will discard all your local changes to that file, so make sure you've saved anything else you wanted to keep first. If you have already pushed, you can view the previous version of any file on GitHub by going to the file → clicking "History" → selecting the previous commit.
The site is down or showing a GitHub 404 page.
Check the following: (1) Log into Namecheap and verify the domain is still active and pointing to GitHub Pages. (2) Go to the GitHub repo → Settings → Pages and confirm that GitHub Pages is enabled and the source branch is set to main. (3) Check if there's a red X on recent commits (visible on the repo's main page) indicating a deployment error — click it for details.
The CSS looks wrong — colors, fonts, or layout are off.
Check the style.css file line count with wc -l style.css in the Terminal (should be ~1,471 lines). A significantly lower number suggests the file was accidentally truncated — restore from Git. Also check that the <link rel="stylesheet" href="style.css"> line is present in the <head> of the affected page and that the path is correct.

Git & GitHub

Git is version control — software that tracks every change ever made to every file in the project, who made it, and when. Think of it like a detailed "track changes" history for the entire codebase going back to the very beginning. You can view any file as it existed on any date, see exactly what changed between two versions, and restore anything that was accidentally deleted.

GitHub is the website that hosts the repository online. Git is the tool running on your computer; GitHub is the shared server everyone syncs to. They are separate things that work together: you use Git commands in your Terminal, and those commands talk to GitHub.

A few core concepts worth understanding before diving into commands:

Repository (repo)
The project folder that Git is tracking. Everything inside brooklyn-alphas/ is the repository. It contains all the site files plus a hidden .git/ folder where Git stores its history — never delete that folder.
Commit
A saved snapshot of the project at a specific moment. Every commit has a message describing what changed and is permanently recorded in the history. Think of commits like save points in a video game — you can always go back to any of them.
Branch
A parallel version of the codebase where you can make changes without affecting the live site. main is the production branch — what visitors see. You create a new branch for each set of changes, work there safely, and merge it into main via a Pull Request when it's ready.
Pull Request (PR)
A proposal to merge your branch into main. GitHub shows a diff — a side-by-side view of exactly what changed — so a reviewer can check the work before it goes live. This is the gate between your edits and the published site. The main branch on this repo is protected, meaning PRs must be approved before merging.
Staging area
A holding area for changes you intend to include in your next commit. git add . moves all your changed files into the staging area. git commit then packages everything staged into a permanent snapshot.
Push / Pull
git push uploads your local commits to GitHub. git pull downloads any commits on GitHub that you don't have locally yet. Always pull before starting work so you're starting from the latest version.
Merge conflict
What happens when two people edit the same line of the same file at the same time and Git can't automatically decide which version to keep. Rare on a small project like this, but possible. VS Code has a built-in conflict resolver that highlights the two versions and lets you choose.

Working in VS Code

VS Code is the recommended code editor for this project. A few features are especially useful when working on this site:

File Explorer
The left panel. Click any file to open it. You can also drag files directly into the panel to add them to the project folder.
Find in File
Ctrl+F / Cmd+F — search within the current file. Essential for finding specific text or class names in long HTML files.
Find in All Files
Ctrl+Shift+F / Cmd+Shift+F — search across every file in the project. Use this when updating nav links or a year that appears on multiple pages.
Find & Replace
Ctrl+H / Cmd+H — find text and replace it with something else. You can also use Ctrl+Shift+H to replace across all files.
Integrated Terminal
Ctrl+` (backtick) opens a terminal panel directly inside VS Code. Use this to run Git commands and Node.js scripts without switching windows.
Live Preview
Install the Live Preview extension (search Extensions panel, Ctrl+Shift+X) to see your HTML file update in real time as you type — no need to refresh the browser manually.
Source Control tab
The Git icon in the left sidebar. VS Code has a built-in Git GUI — you can stage, commit, and push from here without typing terminal commands, once you're set up.
